Phase 1: 子领域扩展

从 6 大领域生成细分研究方向与搜索关键词

▼
输入→6 个硬编码研究大领域→输出 ~30 个子领域（含搜索关键词）
核心逻辑
  • LLM 为每个大领域生成 N 个具体、当前活跃的研究子领域
  • 子领域数量 N 由目标 query 总数反推，保证漏斗产量足够
  • 每个子领域附带 3-5 个 Semantic Scholar 搜索关键词
  • 要求子领域足够具体（如「检索增强生成」而非「NLP」）
🔍

Phase 2: 论文发现与作者定位

通过 Serper (Google Scholar) 搜索论文，Semantic Scholar 补全摘要与元数据

▼
输入→子领域 + 关键词→输出 每个子领域的论文池
两阶段时间窗口
作者发现窗口近 36 个月
论文采集窗口近 18 个月

先用 2-3 年窗口找高引论文（确保引用量充足），提取活跃作者后，再拉取他们近 12-18 个月的最新发表（确保前沿性）。

核心逻辑
  • 多策略搜索：先用组合关键词搜索 → 再用子领域名称 → 最后逐个关键词尝试，确保论文覆盖
  • 两步获取：Serper (Google Scholar) 搜索找到论文列表 → Semantic Scholar 按标题匹配补全完整摘要和结构化作者信息
  • 按引用速率（citations / 论文月龄）排序，避免偏向旧论文
  • 统计高频出现的作者 → 取 top 5 活跃研究者
  • 拉取每位作者的近期发表（通过 S2 API） → 汇总为子领域论文池
关键参数
top 论文 / 子领域20
top 作者 / 子领域5
论文 / 作者10

Phase 3: 知识提取 + Layer 1 验证

从论文摘要中提取结构化知识，并验证提取准确性

▼
输入→论文池（含摘要）→输出 通过验证的结构化知识库
提取内容（LLM Call #1）
  • research_question — 核心研究问题
  • methodology — 方法论概述
  • key_findings — 关键发现列表
  • limitations — 研究局限
  • claims — 具体可验证声明（含类型标签）
Layer 1 验证逻辑（LLM Call #2）
  • 将提取结果与原始摘要交给另一次 LLM 调用做对比审查
  • 通过条件：accuracy ≥ 0.8 且无 critical 级别错误
  • 未通过验证的论文直接移除，不进入后续阶段
🔗

Phase 4: 关系挖掘 + Layer 2 验证

发现论文间的方法论争论、矛盾、互补等非平凡关系

▼
输入→已验证知识库→输出 已验证的论文关系组
五种关系类型
  • 方法论争论 — 不同方法研究同一问题
  • 矛盾解决 — 论文结论相互冲突
  • 文献综合 — 覆盖同一主题的不同方面
  • 定量验证 — 可交叉引用的量化结果
  • 跨领域桥梁 — 不同子领域的方法可互相借鉴
Layer 2 验证逻辑
  • 将检测到的关系连同论文摘要交给 LLM 做二次确认
  • 核心问题：「这些论文是否真的在研究同一问题？矛盾是否真实存在？」
  • 通过条件：verdict = verified 且 confidence ≥ 0.7
✍

Phase 5: 查询生成 + Layer 3 验证

基于论文关系生成高难度研究问题

▼
输入→已验证关系组→输出 Query + 源论文
生成内容
  • query — 30-80 词研究问题（有明确交付物与时间范围）
  • expected_answer_elements — 完整回答应包含的要素清单
  • difficulty_factors — 难度来源分析
Layer 3 验证逻辑
  • 核实查询元数据中归因到特定论文的声明是否与论文摘要一致
  • 检查是否存在编造的数据、方法或错误归因
  • 通过条件：verification_passed = true 或 score ≥ 0.6
🛠

Phase 6: 过滤与质量控制

搜索必要性验证 → 难度下限 → 去重 → 策略均衡

▼
四道过滤门
  • 搜索必要性测试 — LLM 在无搜索条件下回答 → 评估基线质量 → 淘汰基线已足够好的问题（quality_score ≤ 0.7 才保留）
  • 难度下限 — 需引用论文 ≥ 2 篇，难度因子 ≥ 2 项
  • 语义去重 — Jaccard 相似度 > 0.6 视为重复
  • 策略均衡裁剪 — 按权重分配配额，裁剪至目标数量
基线评估五维度
准确性accuracy
完整性completeness
深度depth
时效性timeliness
具体性specificity

核心逻辑：如果 LLM 不搜索就能给出高质量回答 → 该问题不需要深度搜索 → 淘汰。

Practitioner Insights

从律师真实需求到 Pipeline 设计

与科研文献综述不同，法律 Deep Research 的核心特征是没有唯一解 — 不同的解释者、不同的定义会推导出不同答案。任务性质更接近推断和预测，而非检索。以下洞察来自与法律从业者的深度访谈。

⚖ 工作流：三段论推理

先定位 primary source（政府发布的法律法规），再寻找支撑案例。以中国法律体系为例，遵循三段论 — 大前提（法条）、小前提（事实）、结论。大前提本身是确定的，但大前提的解释才是真正的难点。

→

Pipeline 设计响应

Phase 2 严格区分三类法源（primary / case_law / secondary），搜索时优先权威站点；Phase 3 提炼每个法域的"解释空间"作为差异分析的核心维度。

⚠ 核心担忧：Secondary Source 的迷惑性

律师、学者的解读（secondary source）本身带有主观性。AI 如果处理不好，产出的结果"迷惑性很强" — 看似合理实则有误。这是法律从业者对 AI 工具最大的担忧。

→

Pipeline 设计响应

Phase 2 交叉验证要求 secondary source 必须追溯到 primary source；Phase 4"无先例"类 prompt 明确要求基于 primary source 独立推断，而非照搬解读；Phase 5 评估"迷惑性风险"作为 query 质量的正向指标 — 越容易误导模型的问题越好。

📚 三种问题类型各有侧重

有先例 — 找类似 case 作为参照，但案例间的可比性本身就存在争议；无先例 — 如 AI 版权争议，没有现成答案，需从法条推断；预测类 — 最难也最有价值，核心在于"研究人的行为"：法官倾向、利益考量。

→

Pipeline 设计响应

Pipeline 均衡生成三种类型，每种使用专门 prompt 策略。预测类要求考虑"人的变量"，有先例类考察真实案例检索能力（而非编造案例），无先例类考察法条推断能力。
